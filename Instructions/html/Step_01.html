Introduction

Building executable specifications is an effective and proven way to communicate design ideas and documentation of low level requirements. Simulink and Stateflow are ideal platforms for defining executable specifications in embedded software development. In the model-based development process these executable specifications are then elaborated to create production code ready models.

Executable specifications provide an opportunity for early validation of requirements. You can enhance this by providing a link between parts of your model and the corresponding requirements. And you can answer the question: Are all my requirements implemented?

Simulink Requirements provides a requirements editor to enable you to author requirements without an external tool. Simulink Requirements allows you to associate (or link) requirements to the implementation in Simulink models and to verification test cases in Simulink Test.

You can use Simulink Requirements to demonstrate requirements traceability by:

  • Authoring requirements and importing external requirements for linking
  • Creating "implementation" links to Simulink blocks, to Stateflow objects and to lines within a MATLAB Function Block
  • Creating "verification" links to test cases authored in Simulink Test and property proof models in Simulink Design Verifier
  • Creating "derived" links from low level requirements to high level requirements
  • Using the requirements editor (a requirements centric view) to navigate to the implementation and test cases
  • Creating a code generation report with hyperlinks to the requirements

In the workshop, we will use Simulink Requirements to create and view three types of requirement links:

  • Implementation links
  • Verification links
  • Derived links

As we said above, we will answer the question: Are all my requirements implemented? The verification links will answer the question: Does the implementation pass all functional requirements? We will pull ahead the execution of the tests to demonstrate the implementation meets the functional requirements along with the requirements traceability to the test cases. In a later step we will show in detail how to create the test cases.

What requirements are implemented in the model?

Let's first take a look at a set of requirements for our Cruise Control model authored in Simulink Requirements.

Open the requirements "MW_CruiseControl.slreqx" in the Requirements Editor – click here.

In the Requirements Editor we can see (2) levels of requirements:

  1. High level requirements, "SystemReqs" imported from an external spreadsheet
  2. Low level requirements, "MW_CruiseControl" authored in Simulink Requirements

This example has links from the high level requirement to the low level requirements. To navigate to the derived low level requirement:

  • Select the high level requirement "INTF-1: Cruise Control Switch"
  • In the Links pane select "#2 Enable/Disable Switch"

You can also navigate back from the low level requirement to the high level using the same method. In a later section of this step we will show how the high level requirements were imported from a spreadsheet.

The low level requirements in this example have been interpreted into the design as an implementation model. To demonstrate traceability from the requirements to the design implementation, there are links from the low level requirements to the model elements. From the Requirements Editor you can navigate from the requirement to the model object using the implementation link.

To trace the requirement to the implementation, do the following:

  • Select the requirement "Enable/Disable Switch"
  • Click the implementation link "CruiseOnOff" inport in the Cruise Control model

To navigate to the associated requirement,do the following:

  • Right-click on "CruiseOnOff" inport to show the block popup menu
  • Select Requirements Traceability / 1. "Enable/Disable Switch ..." to navigate to the associated requirement

From the previous exercise, by right clicking on a block, you can tell if there is a requirement associated with that block, but what if you just want to get a "bird's eye view”? For this section, we have a model that has multiple links to the requirement document. To get a "bird's eye view” of the blocks with requirement links, we will use the Requirements Perspective feature.

To enter the Requirements Perspective, do the following:

  • In the model, click on the "perspective" badge in the lower right corner of the model to popup the list of perspectives.
  • Select Requirements to enter the Requirements Perspective.

You can now see all the blocks that have requirement links have a requirements badge.

To see the associated links, do the following:

  • Click on any block to see the associated requirement is now highlighted in the Requirements list, or
  • Click on any requirement to see the associated block is now highlighted

To view the links in the state chart, do the following:

  • Open the "Compute target speed" subsystem to see the links associated with states and transitions in the Stateflow chart.
  • Click on transitions and states to see the associated requirement is now highlighted

Note: Model objects can have one or more linked requirements and a requirement may be linked to one or more model objects.

Linking Requirements to Simulink Blocks

In the requirements list, we can see the implementation status for the the requirements by displaying the Implemented column to see if there are requirements without implementation (model) links.

To show the implementation status, do the following:

  • Right-click in the Requirements list to show the popup menu.
  • Click Implementation Status

From the implementation status we can see that (2) of the "Interface" requirements have no implementation links:

  • Vehicle Speed Input
  • Vehicle Brake Input

Return to the top level of the Cruise Control model. From the model, we can also see that the "Speed" inport and the "Brake" inport are not linked as shown by the absence of the "requirements badge".

This may have happened due to missing requirements that were found during the implementation design. These requirements are color coded "green" to show they were created after the implementation.

The next task is to link these ports to the correct new requirement. To link the "Vehicle Speed Input" requirements to the "Speed" inport, drag the requirement to the block. And do the same for linking "Vehicle Brake Input" to the model "Brake" inport.

Verify the links are correct by clicking on the model object and checking the correct requirement is highlighted. You can also select the requirement to see the proper model object is highlighted.

This completes the exercise of linking requirements to Simulink objects. We will now repeat this for Stateflow in the next exercise.

Linking Requirements to Stateflow Objects

Open the "Compute target speed" state chart. Looking at the requirements link badges on the transitions, there were some inconsistencies with regard to the transitions with links and other transitions without links. One inconsistency to note is that both "hold" function transitions are are not linked to requirements.

We are pulling ahead an issue that we will find in a later step to illustrate linking to Stateflow objects.

After reviewing the requirements, it was determined that there were not any requirements for the "hold" function. These have been added to the requirements document to capture the "hold" function. As you can see the (2) new "hold" requirements have no implementation links. We will now link these new requirements to the "hold" transitions.

The same linking mechanism for the Simulink blocks can be applied to Stateflow objects. To link the new "hold" function requirements to the Stateflow "hold" transitions, do the following:

  • Select the requirement: "2.10 Continuous "Accelerate" input increases..."
  • Drag to the transition connected to the "Accel" state, "[after(10/holdrate*incdec,tick)]".
  • Next, select the requirement: "2.11 Continuous "Decelerate" input decreases..."
  • Drag to the transition connected to the "Coast" state, "[after(10/holdrate*incdec,tick)]".
  • Verify the links are correct by clicking on the transitions and requirements to check the associated links.

After you've completed the Statelflow linking task, you can exit the Requirements Perspective by doing the following:

  • Select the "perspective" badge and click Exit Perspective. model.

What requirements are implemented in the data dictionary?

In the previous section we have shown links from the low level requirements to Simulink and Stateflow elements. Another part of the design is the Data Dictionary, a persistent repository to store design data for parameters (calibrations) and signals that adds more definition to the behavior of the implementation model.

To view the linked data dictionary items, do the following:

  • Open the Model Explorer with the data dictionary detailed view by selecting the badge in the lower left corner of the model.

  • To see what requirements are linked to the data dictionary items, right-click on an item to see if there is a reference to a linked requirement. Use this popup menu to navigate from the "maxtspeed" item to the linked requirement.

The Requirements Editor now shows the requirement linked to the data item. In the Links section of the highlighted or selected requirement there are (2) data items linked to this requirement.

  • From the editor, navigate back to the linked data item in the Model Explorer by selecting one of the data item hyperlinks.

Linking Requirements to the Data Dictionary

In the "Functional" section of the requirements, there is a requirement defining the expected behavior when the driver continues to press or "holds" one of the speed change button inputs ("accelerate" or "decelerate"). There is a design parameter, "holdrate" that controls the rate of how quickly the target speed changes during the "hold" event. We will now show how to link a requirement to a design data item by linking the "holdrate" parameter to the "hold" functional requirement:

  • Select the requirement to be linked in the Requirements Editor. In this case, select the requirement Continuous "Accelerate" input increases....
  • Right-click on the requirement to select the menu item, Select for Linking with Requirement.

  • Navigate back to the Model Explorer. Right-click on the "holdrate" data item and select Requirements/Link to selection in Requirements Editor.

  • Once this step has been completed the link has now been created. Check the link by navigating from the data item to the requirement and back to the data item.

This completes the exercise of linking requirements to design data items. We will now explore test case linking in the next sections.

What are the test cases to verify the implementation?

In the previous sections we showed how to create and view the implementation links to the requirements. In this section we discuss how to create and view verification links to the requirements. Functional requirements require some form of verification to demonstrate the implementation behavior is meeting the requirements. Typically this is accomplished with a set of tests that check for the expected response to a stimulus designed to exercise the requirement function.

To show the requirement links to test cases we will return to the Requirements Editor and open the associated Simulink Test file, "CruiseControl_TestCase.mldatx" - click here.

In addition to the implementation link, the requirement also has a verification link to navigate to the test case in Simulink Test.

To navigate to the test case, do the following:

  • Click on the verification link "Init Not Enable or Engaged"

The test case is now displayed in Simulink Test along with the requirement summary hyperlink to navigate back to the associated requirement.

To navigate back to the requirement, do the following:

  • Expand the Requirement field
  • Click on the requirement link "Init Not Enable or Engaged"

As we did before in the Requirements Perspective, we can get an overall status of the implementation and the verification. With the verification status we can easily to see what requirements have linked test cases and the result of the test case.

To show the implementation status and the verification status, do the following:

  • Select the menu item Display/Implementation Status in the Requirements Editor.
  • Select the menu item Display/Verification Status in the Requirements Editor.

Now we can see all the requirements in the "Functional" section have implementation and verification links. Currently the verification status is "unknown" so we need to execute the test cases to update the status.

To update the verification status, do the following:

  • Navigate back to Test Manager
  • Select the "TestSuite"
  • Right-click on the "TestSuite" and select "Run"

Once the tests are completed, press Refresh to update the verification status with the test results.

Another way to update the verification status is to use the Run Tests feature that can be launched by right-clicking anywhere in the requirements set hierarchy.

A set of tests are presented in the Run Tests popup dialog. The tests in the dialog list depends upon where in the requirements set hierarchy the Run Tests feature was invoked. From this dialog you can run all the tests or select a subset of tests to be executed.

Linking to Property Proving Test Harnesses

In the previous sections we showed how dynamic test cases can be used to verify the requirements. Simulink Design Verifier can also be used to verify requirements using behavioral proofs. Using the Property Proving feature of Design Verifier we can prove the implementation model will always meet requirements based on a static analysis.

We will focus on proving the implementation will always satisfy the requirement "Decelerate" input decreases target speed only when engaged (active)

To see how we can use this new method, do the following:

  • Open the Property Proving harness - click here.

  • Open the "Verification Subsystem" to inspect the requirements behavior model.

The Proof block "Prop_coast_mintspeed" is used to determine the results of the analysis. It is linked to the requirement as a "verification" link type.

  • Right-click the Proof and navigate to the requirement specified at the top of the submenu.

  • Notice the "Prop_coast_minspeed" verification link from the Proof block to the requirement. Once the Property Proving analysis is complete the link will show the verification status similar to the dynamic testing verification links to Simulink Test.

Demonstrating Traceability to the Code

Now that we have good traceability from the requirement to the model object. Let’s work through a method that will demonstrate the complete traceability from requirement to model to source code.

Save the implementation links in the model by Analysis/Requirements/Links File/Save Links so our changes will be used during code generation

To generate the code, do the following:

  1. Under Code, select C/C++ Code, and Build Model. Note: the options for embedding the requirement comments in the C source code have already been set.

After the code has been generated (should be 5-10 sec), the following code generation report should open:

Notice the links to the model blocks embedded in the code comments.

  • Click on one of the '<Root>/Brake' links to highlight the "Brake" inport block in the model.

Next, we are going to use the Navigate to Code function to return from the model to the code.

  • Right-click on the highlighted "Brake" inport block
  • Select C/C++ Code and Navigate to C/C++ Code.

This should take you back to the code generation report with the code comment associated with that block highlighted.

Now we will navigate to a requirement using hyperlinks in the code generation report:

  • Click on "1. Vehicle Brake Input" to navigate to the associated requirement.

The Requirements Editor is displayed with the associated requirement highlighted.

We can also trace from the Requirements Editor directly to the generated code.

  • To enable tracing from the requirement to the code, select the menu item Analysis/Show code traceability.

  • To trace to the linked code from the requirement, select the "code" icon in the implementation link.

In the generated code report, there are hyperlinked comments for the linked data items that includes the name of the data dictionary and the data item.

With these functions shown above, you can go from Requirement to Model to Source Code, or directly from Source Code to Requirement to demonstrate requirements traceability.

Handling a Requirements Change

In our example there was a set of read-only high level requirements that had been imported from a spreadsheet into the Requirements Editor. In this section we'll show how to handle a change to these external high level requirements.

First we will update the external requirements by copying over the existing requirements spreadsheet with the new requirements spreadsheet. To do the copy and view the updated, external high level requirements spreadsheet "SystemReqs.xlsx" - click here

Check the first requirement "INTF-1" in the external requirements spreadsheet. Notice it has been changed to now include a fourth button, "Cancel". To handle the requirement change do the following:

  • Let's use the update feature in Simulink Requirements to import the requirement change.

  • Once the update has completed notice there is a summary of changes in the Comments section that indicates the "INTF-1" has been updated or changed.

  • There is a highlighting feature to show possibly impacted requirements. To highlight the changed requirement select Display/Change Information.

  • Navigate to the changed requirement "INTF-1" to investigate the impact of the change which shows (3) linked requirements in the "Interface" section of the lower level requirements may be impacted.

  • As described in the tooltip, to clear the issue open the Links view.

The requirements with Change Issues will not require any further handling so we can clear these change issues but we will need to add requirements to include the "Cancel" button in the "Interface" section and add a requirement to describe the functional behavior in the "Functional" section.

  • In the Links view we can clear the issues by right-clicking at the links file level and select Clear All Change Issues.

To add the (2) new requirements, it is assumed that you can intuitively add these requirements based on the popup menus in the editor and link the new requirements to the model as we did in the previous sections.

Or you can load the requirement set and model changes based on the new "Cancel" button from the "INTF-1" change - click here

  • Based on requirements changes based on the discussion above, note the (2) new requirements added to the "MW_CruiseControl" requirement set.

  • We also linked the (2) new requirements to the (2) model modifications required to implement the new requirements. Check these links in the Requirements Perspective.

Importing High Level Requirements

In our example there was a set of read-only high level requirements displayed in the Requirements Editor. In this section we'll show how these high level requirements were imported from a spreadsheet.

To open the high level requirements spreadsheet "SystemReqsImport.xlsx" - click here

To import the spreadsheet into the Requirements Editor do the following:

  • Close (if open) the existing high level requirements opened earlier in the step

  • Select File/Import to open the import dialog

  • In the Importing Requirements dialog, select "Microsoft Excel Spreadsheet" in Document Type
  • Browse to select "SystemReqsImport.xlsx" in Document Location
  • Select Configure Columns

  • In the Configure columns dialog, assign the attributes to the column header as shown below

  • Select OK to use the column settings
  • Select Import in the Importing Requirements dialog to begin the import

With the imported high level requirements, we will link a high level requirement to the model design low level requirements.

Do the following to create the link:

  • In the Reguirements Editor, for the "INTF-1" high level requirement Select for Linking With Requirement

  • Navigate to the "Enable/Disable Switch" requirement in the "MW_CruiseControl" set
  • Right-click and select * Create a link from "INTF-1: Cruise...*

We now have a link from the imported high level requirement to the low level design requirement authored in Simulink Requirements.

Importing Requirements from Word

In the previous section we showed how to import high level requirements from a spreadsheet. In this section we will show how to import requirements from a Word document. For this example we will import an existing set of low level requirements similar to the requirement set we used to link to the model and test cases.

To open the low level requirements document "cruise_control_reqsID.docx" - click here

Notice each requirement is identified as "RQnnn". We will use this to import the requirements and use this as the "ID" attribute.

To perform the import do the following:

  • Select File/Import to open the import dialog

  • In the Importing Requirements dialog, select "Microsoft Word" as Document Type
  • Select Browse to select the document "cruise_control_reqsID.docx"
  • Select the option Rich text (include graphics and tables) from the document
  • Select the option Identify items by occurrences of search pattern (REGEXP)
  • Enter the search pattern "RQ\d{3}" to import the items

  • Select Import in the Importing Requirements dialog to begin the import

The requirements are now visible in the Requirements Editor and linking can be established to the requirements using the same steps performed in previous sections.

  • Generated code traceability from the code to the model and requirements

So going forward we have confidence that our model has implemented all of the requirements.
